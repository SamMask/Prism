戰略建議：面對「上萬筆」的正確規劃
SQLite 的效能遠比你想像的強大。幾萬筆純文字對 SQLite 來說是「小菜一碟」。

不用做 B，也不用做 A。請採用 「方案 D：SQLite 效能調優 (WAL Mode + FTS5)」。

這是針對 Local Insight 的長期規劃建議：

步驟一：保持現狀 (方案 C 架構) ✅
繼續把文字存 DB，圖片存 static/uploads。

理由： 純文字非常小。1 萬字大約才 30KB。存 10,000 篇長文，資料庫也才 300MB 左右。這對現代電腦來說根本無感。

步驟二：開啟 WAL 模式 (Write-Ahead Logging) ⚡ (關鍵!)
這是 SQLite 效能起飛的開關，支援高併發讀寫。

原理： 寫入時不鎖死整個資料庫，而是先寫入 log 檔，讀取可以同時進行。

實作： 在 app.py 的資料庫連線設定加一行：

Python

conn.execute('PRAGMA journal_mode=WAL;')
步驟三：FTS5 全文檢索 (你已經做了) 🔍
你已經規劃了 FTS5。這是處理上萬筆資料搜尋秒回的唯一解。

注意： 確保 FTS 的 Trigger 有寫好，資料更新時索引要同步。

步驟四：封存機制 (Archiving) - 這才是你要的 "分表"
如果未來真的有 10 萬筆資料，且覺得列表載入變慢：

設計： 新增一個 is_archived (Boolean) 欄位。

邏輯： 預設查詢 WHERE is_archived = 0。

效果： 舊資料還在同一個表裡（方便搜尋），但不會卡住日常的列表顯示。

總結與決策
方案 A (時間分表)： ❌ 廢棄。不適合筆記應用。

方案 B (存 .md 檔)： ❌ 不建議。除非你要做「雙向同步 Obsidian」，否則會增加大量檔案 I/O 的複雜度。

方案 C (附件分離)： ✅ 保持。你現在就是這樣做的，這是對的。
