## 1. 執行摘要 (Executive Summary)

**Local Insight** 目前已具備 MVP (最小可行性產品) 的核心能力，成功實現了「零依賴、本機執行」的輕量化目標。其 Python/Flask + SQLite + Vue.js 的架構選擇極為精準，兼顧了開發效率與使用者部署的便捷性。

## 2. 現有架構評估 (Current Architecture Review)

### ✅ 優勢 (Pros)

- **極致輕量**: 堅持不引入 Node.js build process 與 Redis 等外部依賴，使得專案可攜性極高（USB 隨插即用等級）。
- **離線優先**: 前端資源完全本地化，符合「去中心化」與「隱私安全」的核心價值。
- **視覺基礎**: 使用 Tailwind CSS 建立了深色模式與基礎的瀑布流介面，具備現代化 App 的雛形。

### ⚠️ 風險與技術債 (Risks & Tech Debt)

- **搜尋瓶頸**: 目前使用 SQL `LIKE %keyword%` 進行模糊搜尋。當筆記數量增長至 1,000+ 篇時，效能將顯著下降，且不支援語意或模糊容錯。
- **資料單點風險**: 雖然有單檔資料庫備份，但缺乏「單一筆記的歷史版本」。使用者若誤刪重點提示詞內容並儲存，將無法回溯。
- **互動細節**: 瀑布流目前依賴 CSS Grid，卡片高度不一時可能會產生視覺空隙 (Gaps)。此外，提示詞管理過於「靜態」，缺乏「一鍵複製」、「快速試驗」等提升效率的微互動。

## 3. 改善路徑 (Improvement Roadmap)

### 3.1 技術底層優化 (Technical Optimization)

#### **T1. 引入 SQLite FTS5 全文檢索 (High Priority)**

- **現狀**: `WHERE content LIKE ?` (掃描全表)。
- **建議**: 啟用 SQLite 的 **FTS5 (Full-Text Search)** 模組。
- **效益**: 搜尋速度提升 10-50 倍，並支援 token 搜尋（如輸入 "py flask" 可找到同時包含兩者的筆記），大幅提升知識檢索效率。

#### **T2. 圖片懶加載 (Lazy Loading) 與壓縮**

- **現狀**: 瀑布流一次載入所有圖片，可能導致啟動時記憶體飆升。
- **建議**:
  1. 前端 `<img>` 標籤加入 `loading="lazy"` 屬性。
  2. 後端上傳時加入 `Pillow` 函式庫進行圖片自動壓縮與 WebP 轉換（在不犧牲太多畫質下減少 70% 體積）。

#### **T3. 增強流暢度的 View Transitions API** //PM 備註：此處重要

- **現狀**: 切換 Type 或 Tag 過濾時，卡片是瞬間消失/出現。
- **建議**: 利用 Vue 的 `<TransitionGroup>` 配合 CSS transform，實現卡片「洗牌 (Shuffle)」的動畫效果。
- **商業價值**: 這種「絲滑」的視覺回饋是區分「工具」與「精品 App」的關鍵細節。

### 3.2 使用者體驗優化 (UX Polish) //PM 備註：此處重要

#### **U1. 提示詞專屬微互動 (Prompt UX)** //PM 備註：此處重要

- **痛點**: 目前複製提示詞可能需要點開卡片 -> 選取文字 -> 複製。
- **建議**: 在卡片表面 (Hover 狀態) 新增 **「一鍵複製」按鈕**。若卡片類型為「提示詞」，點擊卡片可預設行為改為「複製內容並顯示 Toast 提示」，而非開啟編輯器。

#### **U2. 移動端適配深化 (Mobile First)** //PM 備註：不增加移動端，如果需要的話等有上線網站有手機版再說

- **建議**: 增加觸控手勢支援。例如在列表模式下，**左滑卡片**可觸發「快速歸檔」或「刪除」，**右滑**觸發「編輯」。

## 4. 新功能提案 (New Feature Proposals)

### 4.1 商務/功能面 (Business Features)

#### **F1. 筆記時光機 (Version History) ⭐⭐⭐** //PM 備註：此處重要，需歷史記錄

- **描述**: 每次儲存 (`PUT`) 時，不直接覆蓋，而是先將舊內容寫入 `Note_History` 表 (Diff 或是完整快照)，再更新主表。

  2. **迭代對比**: 對於 AI Prompt 調試，使用者常需要對比 "v1" 與 "v2" 哪個效果好，歷史紀錄至關重要。
