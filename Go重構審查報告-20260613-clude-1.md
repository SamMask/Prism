## 一、總體判斷

1. **Go primary 本身是真實且大致完整的實作，不是半成品空殼。** `go-shadow/main.go` 有 8811 行、321 個函式，前端 `api.ts` 與直接 `fetch()` 呼叫的每一條 `/api/*` 路由都有對應 Go handler，無一漏接。
2. **無殘留 stub / TODO / placeholder / not-implemented**：Go 原始碼掃描沒有真正的佔位程式（grep 命中的 `placeholders` 全是 SQL 參數輔助函式，誤判）。
3. **核心功能在 Go 端皆已落地**：notes CRUD/actions/history/batch、FTS 搜尋、attachments metadata + raw serving、upload / upload-url / upload-delete / extract-prompt、長文 separation/restore、media cleanup、import/export（json/markdown/db/images/batch）、categories、tags、prompt/wizard options、server/system/backup。
4. **最大風險不在「漏接」，而在「驗收的真實性」**：約 **58/81 個 pytest 檔案是文件治理測試**（只 `read_text` / `json.loads` 然後斷言 markdown 或 contract JSON 裡的字串），不執行任何 runtime。525 passed 的數字被這類自我指涉測試大幅灌水。真正跑 Go runtime 的功能 parity 測試只有 23 個檔案。
5. **功能性 pytest 測的是 Python（Flask）backend，不是 Go**：`tests/conftest.py` 用 `from app import create_app` 建 Flask test client。Go 的正確性靠「parity 測試（Go vs Python diff）+ `go test`」間接保證——這套邏輯合理，但代表 **Go-only 行為沒有獨立的端對端 pytest 覆蓋**。
6. **Pi「已 live cutover」的宣稱無法從 repo 驗證**：T042-T044 狀態為 Done，但其 pytest 測試只讀 contract JSON；真正的執行證據（`build/go-primary-live/pi/evidence.json`）落在 gitignore 的 `build/` 下，不在版控快照內。操作腳本 `scripts/go_primary_pi_live_ops.ps1` 本身是完整且合理的。
7. **唯一明確未完成的 active 項目是 T053**（Python backend source 最終封存/刪除 + docs/API/release 文案收斂），文件已誠實標示為 Todo。
8. **文件與實作有少量過期不一致**（README 仍說測試 fixture 走手寫 schema，但 `conftest.py` 已改走真實 migration chain），屬低風險。
9. **根目錄整體可控**：主要雜物（`garbage-can/`、`app.log`、`test_run.log`、`build/`）皆已 gitignore，未污染版控；少數歷史報告散落在 `docs/過期`、`docs/倉庫`。
10. **建議**：可以進入收尾修復階段，但**第一優先不是改 code，而是補上 Go-only 端對端驗收**，並執行（或標示未執行）T053。現在不該做任何新功能或大重構。

## 三、Go backend / API contract 檢查

### 已確認正常的部分（前端呼叫 → Go handler 全對齊）

- **路由註冊**（`go-shadow/main.go:181-225`）涵蓋 categories、tags、notes、attachments、system（migration-status / check-update / stats / vacuum / clear-history / startup-preference / wal-checkpoint / check-consistency / port-config）、server（hardware / logs / restart / version / backup list/download/rotate/delete）、prompt-options、wizard-options、cleanup（orphan-images / originals / broken-images）、export（json/markdown/db/images）、import/json、upload（/ url / delete / extract-prompt）、靜態 SPA。
- **notes 子路由 dispatcher**（`main.go:7881-8116`）完整處理 `import/md`、`export/batch`、`reorder`、`batch/*`、`<id>/attachments`、`<id>/pin|archive|duplicate`、`<id>/history`（GET/DELETE）、`<id>/check_separation`、`<id>/separate`、`<id>/restore`（含 separation restore 與 `restore/<historyId>` 版本還原雙語意）。
- **前端 separation 三連呼叫**（`api.ts:457-488`：`check_separation` / `separate` / `restore`）皆有 Go 實作，且 `separateContent`（`main.go:8139+`）以交易處理 auto-extracted attachment upsert + content preview 改寫，邏輯完整。
- **啟動旗標全開**：`scripts/start_go_primary.ps1:54-71` 與 Pi 服務 `scripts/go_primary_pi_live_ops.ps1:271` 的 `ExecStart` 皆帶齊 14 個 `--enable-*` 寫入旗標。**「寫入路由藏在預設關閉旗標後」的疑慮不成立**——production 啟動入口確實全開。
- **嵌入式 SPA**：`//go:embed web/dist/*`（`main.go:66-67`），build 腳本（`scripts/build_go_runtime.ps1`）先 `npm run build` 再把 `frontend/dist/*` 複製進 `go-shadow/web/dist`，兩處 dist 時間戳一致（Jun 13 04:43）。
- **`go test ./...` 通過**（我實際執行，`ok prism-go-shadow`）。

### 疑似漏接的部分

- **無明顯前端→Go 漏接路由**。我比對 `frontend/src` 全部 `/api/*` 字面值與 `api.ts` 的 axios 呼叫，逐條都有 Go handler。T046-T050 已把先前審查列出的四個漏接（extract-prompt、separation、check-update、wizard-options 收斂）補齊。

### 舊 backend / mock / stub / placeholder 殘留

- **Python Flask source 全棧仍在版控**（`app.py`、`routes/`、`services/`、`utils/`、`migrations/`、`templates/`、`static/`、`db.py`、`config.py`、`conftest.py`）——這是 T053 前刻意保留的 legacy/dev/test context，文件已聲明，**不算 bug**，但構成最大一塊待收尾面。
- `routes/system.py` + `app.py` 的 `go_read_routing`（Phase 19 read-shadow proxy）為唯一仍會在測試期產生雜訊（fallback warning）的 legacy 殘件。
- Go 端**無** mock/stub/hardcoded fallback（已掃描）。

### 需要補測的 API（純 Go、不依賴 Python 對照）

- notes CRUD + FTS 搜尋（含文字附件 body search）
- upload / upload-url 的 SSRF/magic/size 邊界（目前靠 Python parity）
- import/json 與 export（db/images）的 round-trip 可還原性
- migration runner 對既有 DB 的冪等升級
- backup rotate/download 路徑安全

> 上述目前皆有 parity 測試覆蓋，但 parity 依賴 Python 對照組；T053 移除 Python 後需要等價的純 Go 版本。

## 四、Frontend / UX 檢查

> ⚠️ 以下為靜態讀碼推測，**未實際啟動前端點擊驗證**，請當作待人工確認的候選清單。

### 可能影響實際使用的 UX 問題（需人工確認）

- **直接 `fetch()` 繞過統一錯誤攔截**：`src/hooks/usePromptBuilder.ts`（`/api/prompt-options`、`/api/wizard-options`、`/api/categories`、`/api/notes`）與 `src/pages/SettingsPage.tsx:88`（`/api/test`）用原生 `fetch`，不走 `api.ts` 的 axios interceptor。若這些呼叫失敗，全域錯誤/Toast 可能不會觸發，使用者只看到靜默失敗。**建議檢查這幾處是否有各自的 catch + 使用者可見提示**（最小改善，不需重構）。

### 錯誤提示 / loading / empty state（現況）

- `frontend/src/pages` + `components` 中有 24 個檔案使用 `isLoading` / `empty` / `catch` / `toast` 類關鍵字，覆蓋面看似合理。
- 無法在不啟動前端的情況下判斷各頁 empty/loading/error 的實際品質（例如搜尋無結果、附件上傳失敗、import 衝突的提示文案）。建議若要收 UX，**先以實機點過一輪**再決定最小改善點，而非憑程式碼臆測。

### 不建議現在處理的純美化項目

- 任何 redesign、版面重排、動畫、主題微調——與 Go 收尾無關，明確暫緩（已符合 `docs/FRONTEND-REDESIGN-PLAN.md` 的暫緩 scope）。

## 五、測試與驗收缺口

### 現有測試狀態

- **Go**：`go test ./...` → 通過（我實際執行）。`go-shadow/main_test.go` 86KB，涵蓋 prompt extraction、separation/restore、check-update、static config 404 等。
- **pytest**：依 `test_run.log`（2026-06-13 13:08）為 **525 passed / 0 failed / 0 skipped**（我未重新執行）。
- **結構性問題**：~58/81 pytest 檔案為文件治理測試（不執行 runtime）；真正跑 Go 的功能 parity 測試為 23 檔；功能性 pytest 跑的是 Python 而非 Go。

### 缺少的核心測試

1. **純 Go 端對端驗收**（不依賴 Python 對照組）——T053 後驗收鏈會斷。
2. **Go-only 行為的回歸**（例如 Go 特有的 WebP thumbnail、embed SPA fallback、`/static/config/*` 回 JSON 404 而非 SPA HTML）目前散在 `main_test.go`，但無一份「核心 workflow smoke」綁定 release。
3. **前端與 Go 的合約整合測試**（目前只有靜態 route coverage contract，無實跑前端打 Go 的整合）。

### 最小建議補測清單

- [ ] 一支「Go-only full workflow」pytest：對 Go HTTP（非 Flask）跑 create → upload → search → separate → export → import → delete → backup → migration，斷言 DB/files invariant。可直接改寫 `scripts/go_primary_full_workflow_smoke.py` 為 pytest 收編。
- [ ] 將既有 parity fixture 的「Go 觀測值」固化成 golden，讓 Python 移除後仍可比對。
- [ ] （可選）為 `usePromptBuilder` 的直接 fetch 失敗路徑補前端單元測試。

## 七、建議修復順序（單一主線）

> 嚴格依「先保資料安全 → 再修整合漏接 → 再補測試 → 再整理根目錄 → 最後 UX 微調」。

1. **保資料安全（P0，最先）**：在動任何刪除前，確認 `knowledge.db` / `backups/` / uploads / attachments / notes 完整且有最新備份。本次審查未發現資料層 bug，此步是 T053 前置保險。
2. **修整合漏接（P1）**：補一支**純 Go 端對端 workflow pytest**（不依賴 Python 對照），把 Go 的核心 workflow 驗收從「靠 Python parity」轉為「自證」。這是 T053 安全執行的前提。
3. **補測試（P2）**：固化 parity 的 Go golden、明確標註「功能測試 vs 文件治理測試」數量，讓驗收完成度不再被 525 這個數字誤導。
4. **執行 T053（P2/P3）**：在步驟 2 的安全網就位後，才封存/刪除 Python source 並收斂 docs/API/release 文案（含修正 README 過期警語、go_read_routing 殘留）。
5. **整理根目錄（P3）**：刪空 `app.log`、決定 `docs/過期`/`docs/倉庫`/一次性 scripts 去留（逐一確認引用，不憑名字刪）。
6. **UX 微調（P3，最後）**：先實機點一輪，再處理 `usePromptBuilder` / SettingsPage 直接 fetch 的錯誤提示等最小改善。
