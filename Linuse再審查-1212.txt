問題本質:

-- Option B: 維持雙欄位 + 強制同步
CREATE TRIGGER sync_type_on_category_update 
AFTER UPDATE ON Categories BEGIN
    UPDATE Notes SET type = NEW.name WHERE category_id = NEW.id;
END;

🔴 不必要的複雜性: 問題 1: 分頁查詢的 N+1 陷阱
# crud.py:141-168 - 使用子查詢而非 JOIN
SELECT n.*,
    (SELECT json_group_array(...) FROM Note_Tags nt WHERE nt.note_id = n.id) as tags_json,
    (SELECT json_group_array(...) FROM Source_Urls s WHERE s.note_id = n.id) as urls_json
FROM Notes n
LIMIT 20 OFFSET 0
為什麼這很糟?
對於每條筆記,SQLite 都要執行 2 個子查詢
當 LIMIT 20 變成 LIMIT 100 時,會執行 100 × 2 = 200 次子查詢
正確的做法是用 LEFT JOIN + 結果集去重
修復方案 (未來優化):
-- 用 CTE + GROUP BY 取代子查詢
WITH tagged_notes AS (
    SELECT note_id, json_group_array(json_object('id', t.id, 'name', t.name)) as tags_json
    FROM Note_Tags nt JOIN Tags t ON nt.tag_id = t.id
    GROUP BY note_id
),
notes_with_urls AS (
    SELECT note_id, json_group_array(url) as urls_json
    FROM Source_Urls
    GROUP BY note_id
)
SELECT n.*, 
       COALESCE(tn.tags_json, '[]') as tags_json,
       COALESCE(nu.urls_json, '[]') as urls_json
FROM Notes n
LEFT JOIN tagged_notes tn ON n.id = tn.note_id
LEFT JOIN notes_with_urls nu ON n.id = nu.note_id
ORDER BY n.updated_at DESC
LIMIT 20 OFFSET 0;

def _cleanup_note_images(content, cover_image):
    # 刪除 cover_image
    if cover_image and cover_image.startswith('/static/uploads/'):
        filepath = os.path.join(upload_folder, filename)
        if os.path.exists(filepath):
            os.remove(filepath)  # ❌ 沒有錯誤處理!
    
    # 用正則提取 content 中的圖片
    matches = re.findall(r'/static/uploads/([^\s\)\]"\']+)', content)
    for filename in matches:

致命缺陷:
沒有引用計數檢查 - 如果兩則筆記共用同一張圖片,刪除其中一則會導致另一則筆記破圖
檔案 I/O 失敗時會靜默吞掉錯誤 (只 print warning)
正則表達式可能誤刪 - 如果 content 包含其他筆記的連結,會誤判為當前筆記的圖片
修復方案:
def _cleanup_note_images(content, cover_image):
    """刪除筆記的圖片,但只刪除沒有其他筆記引用的檔案"""
    db = get_db()
    
    # 收集所有要檢查的圖片路徑
    image_paths = set()
    if cover_image and cover_image.startswith('/static/uploads/'):
        image_paths.add(cover_image)
    if content:
        matches = re.findall(r'/static/uploads/([^\s\)\]"\']+)', content)
        image_paths.update(f'/static/uploads/{m}' for m in matches)
    
    for img_path in image_paths:
        # 檢查是否有其他筆記引用此圖片
        ref_count = db.execute('''
            SELECT COUNT(*) FROM Notes 
            WHERE (cover_image = ? OR content LIKE ?) AND id != ?
        ''', (img_path, f'%{img_path}%', note_id)).fetchone()[0]
        
        if ref_count == 0:
            # 沒有其他引用,安全刪除
            filepath = os.path.join(upload_folder, img_path.replace('/static/uploads/', ''))
            try:
                if os.path.exists(filepath):
                    os.remove(filepath)
            except OSError as e:
                # 記錄錯誤但不阻斷刪除筆記的操作
                print(f"[ERROR] Failed to delete {filepath}: {e}")
BUT - 這個 bug 只在以下情況觸發:
用戶手動複製其他筆記的圖片連結
然後刪除原始筆記
發生機率低,但後果嚴重 (資料損壞不可逆)。建議在 v1.2 修復。
🟡 P1 級風險: WAL 模式的 Checkpoint 管理缺失
# db.py:30
g.db.execute('PRAGMA journal_mode = WAL')
問題:
啟用 WAL 後,資料庫的變更會寫入 .db-wal 檔案
如果從未執行 PRAGMA wal_checkpoint,WAL 檔案會無限增長
當用戶複製 knowledge.db 備份時,如果沒有一併複製 .db-wal,會遺失未合併的資料
修復方案:
# app.py 啟動時加入定期 checkpoint
from apscheduler.schedulers.background import BackgroundScheduler

def checkpoint_wal():
    with app.app_context():
        db = get_db()
        db.execute('PRAGMA wal_checkpoint(TRUNCATE)')
        db.commit()

scheduler = BackgroundScheduler()
scheduler.add_job(checkpoint_wal, 'interval', hours=1)
scheduler.start()
