## 🔴 致命問題 (Critical Bugs)

### Bug #1: 資料庫連線在 `app.py` 中重複定義

**位置**: `app.py` Line 186-201 vs `db.py` Line 14-48

```python
# app.py 中的 get_db() - 這是「死程式碼」
def get_db():
    if 'db' not in g:
        ...
        # 使用 isolation_level=None?? 不，這裡沒有設
        g.db.execute('PRAGMA journal_mode = WAL')
    return g.db

# db.py 中的 get_db() - 這是「活程式碼」
def get_db():
    if 'db' not in g:
        ...
        isolation_level=None  # autocommit 模式
        g.db.execute('PRAGMA foreign_keys = ON')
        # 還有驗證！
```

**問題**: 
- 兩個 `get_db()` 函式，語意不同（一個有 isolation_level 設定，一個沒有）
- 如果誰不小心 import 錯的，Foreign Keys 就不會啟用
- 這是典型的「雙重事實 (Dual Source of Truth)」問題

**【品味評分】**: 🔴 垃圾

**修復方向**: 
刪除 `app.py` 中的 `get_db()`，它只在 `init_db()` 內部使用，應該直接使用 `db.py` 的版本。

### Bug #2: 批次刪除的競態條件

**位置**: `frontend/src/stores/appStore.ts` Line 193-213

```typescript
deleteSelectedNotes: async () => {
    for (const id of selectedNoteIds) {
        await api.deleteNote(id)  // 🔴 一個一個刪！
    }
    // ...
}
```

**問題**:
- N 個筆記 = N 次 HTTP 請求
- 如果第 3 個失敗，前 2 個已經刪了，但 UI 狀態不一致
- 沒有使用後端的批次刪除 API (如果有的話)

**【品味評分】**: 🟡 湊合

**修復方向**:
1. 後端應該提供 `DELETE /api/notes/batch` API
2. 前端一次性發送所有 ID
3. 或者至少用 `Promise.allSettled()` 並處理部分失敗

### Bug #3: 嵌入式 Embedding 線程的資源洩漏

**位置**: `routes/notes/crud.py` Line 44-115

```python
def _queue_embedding_update(note_id: int, title: str, content: str):
    def _do_embedding():
        conn = sqlite3.connect(Config.DATABASE)  # 新連線
        try:
            # ... 操作 ...
        finally:
            conn.close()  # 看起來有關閉...
    
    thread = threading.Thread(target=_do_embedding, daemon=True)
    thread.start()
    # 🔴 沒有任何限制！可以無限產生線程！
```

**問題**:
- 每次儲存筆記都會產生一個新線程
- 如果使用者快速編輯 100 篇筆記 = 100 個線程同時運行
- 沒有線程池 (ThreadPoolExecutor) 做併發控制
- 沒有佇列深度限制

**【品味評分】**: 🔴 垃圾

**修復方向**:
```python
from concurrent.futures import ThreadPoolExecutor

_embedding_executor = ThreadPoolExecutor(max_workers=2, thread_name_prefix="embedding")

def _queue_embedding_update(...):
    _embedding_executor.submit(_do_embedding)  # 最多 2 個同時運行
```

### Bug #4: AI 批次任務的記憶體洩漏

**位置**: `routes/ai.py` Line 303

```python
_batch_tasks = {}  # 🔴 全域字典，永遠不會清理！
```

**問題**:
- 每次執行批次標籤都會在 `_batch_tasks` 新增一個 entry
- 永遠不會被刪除
- 長期運行的伺服器最終會 OOM

**【品味評分】**: 🔴 垃圾

**修復方向**:
```python
import time
from collections import OrderedDict

_batch_tasks = OrderedDict()
MAX_TASKS = 100
TASK_TTL = 3600  # 1小時

def _cleanup_old_tasks():
    now = time.time()
    # 刪除過期或超過數量限制的任務
    while len(_batch_tasks) > MAX_TASKS:
        _batch_tasks.popitem(last=False)
```

## 🟡 架構問題 (Medium Severity)

### Issue #5: `type` 與 `category_id` 的雙重事實

**位置**: `routes/notes/crud.py` 全文

你們有以下情況：
- `Notes.type` - 文字型態 (如 "筆記 | Note")
- `Notes.category_id` - FK 到 Categories

然後你們到處這樣：
```python
type_name = data.get('type', '筆記')
category_id = get_category_id_by_name(db, type_name)
...
UPDATE Notes SET type = ?, category_id = ?
```

**問題**:
- 這是經典的「資料正規化違規」
- `type` 欄位是冗餘的，應該只需要 `category_id`
- 所有「同步 type 與 category_id」的程式碼都是不必要的複雜度

**【品味評分】**: 🟡 湊合 (我理解這是為了向後相容)

**Linus 式方案**:
```
Phase 1: category_id 成為唯一事實來源 (Single Source of Truth)
Phase 2: type 欄位標記為 deprecated
Phase 3: 一年後刪除 type 欄位
```

### Issue #6: 測試 conftest.py 中硬編碼的 Welcome Note

**位置**: `tests/conftest.py` Line 170-172

```python
INSERT INTO Notes (title, content, type) VALUES ('Welcome Note', 'Welcome to Prism!', '筆記');
INSERT INTO Note_Tags (note_id, tag_id) VALUES (1, 1);  # 🔴 假設 ID 是 1！
```

**問題**:
- 如果之後 temp_db 的 INSERT 順序改變，這個假設就會爆炸
- 應該用 `last_insert_rowid()` 獲取實際 ID

**【品味評分】**: 🟡 湊合

### Issue #7: VectorStore 單例模式的 Cold Start 問題

**位置**: `services/vector_store.py` Line 71-77

```python
class VectorStore:
    _instance = None
    
    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
        return cls._instance
```

**問題**:
- `refresh_from_db()` 需要 Flask app context
- 如果在 app context 外呼叫（如測試中），會爆炸
- 單例模式讓測試變得困難（狀態共享）

**【品味評分】**: 🟡 湊合

**建議**: 考慮使用 Flask 的 extension 模式，或者依賴注入。

### ✅ 遷移系統設計

## 📊 五層分析總結

### 第一層：資料結構分析
- **問題**: `Notes.type` 與 `Notes.category_id` 冗餘
- **問題**: `_batch_tasks` 全域字典無限增長

### 第二層：特殊情況識別
- **問題**: `app.py` 和 `db.py` 中重複的 `get_db()` 函式
- **問題**: `parent_id` 欄位的動態偵測 (`has_parent_id`) 增加複雜度

### 第三層：複雜度審查
- **問題**: `_queue_embedding_update` 沒有合理的併發控制，是定時炸彈

### 第四層：破壞性分析
- **風險**: 大規模修改需要小心向後相容性
- **風險**: 測試中的硬編碼 ID 假設

### 第五層：實用性驗證
- **✅ 務實**: 使用 SQLite BLOB 儲存向量，不引入複雜的向量資料庫
- **✅ 務實**: CSRF 用簡單的 Origin 檢查，不需要 token

## 【核心判斷】

✅ **值得做**: Prism V2 是一個有潛力的專案，架構基礎紮實，但有幾個定時炸彈需要立即處理。

### 關鍵洞察

- **資料結構**: 核心問題在於 `type` 與 `category_id` 的冗餘，這導致了大量「同步」程式碼
- **複雜度**: Embedding 和批次任務的非同步處理缺乏合理的資源控制
- **風險點**: 全域狀態 (`_batch_tasks`, 重複的 `get_db()`) 是最大的技術債

### Linus 式方案

1. **立即修復** P0 問題 - 加入 ThreadPoolExecutor 和任務清理機制
2. **統一** `get_db()` - 刪除 `app.py` 中的版本
3. **長期規劃** 移除 `type` 欄位，只保留 `category_id`

## 📁 從 garbage-can 文檔發現的額外洞察

在查閱 `garbage-can/` 資料夾中的歷史文檔後，我發現了幾個值得關注的點：

### ⚠️ 規劃中但未實現的安全機制

**來源**: `999-個人特化版上線前的絕對防禦清單.txt`

文檔提到 `.gitignore` 應包含：
- `knowledge.db` (最重要的個人資料)
- `models/` (本地 AI 模型)
- `embeddings/` (向量索引)

**潛在風險**: 目前的實作中，`embeddings` 實際上存在 SQLite 的 `Embeddings` 表中，而不是獨立的目錄。這是好的設計決策（保持 Portable）。但需確保 `.gitignore` 與實際架構一致。

### 🔍 福爾摩斯的洞察 (來自 998)

福爾摩斯式分析提出了幾個我認同的觀點：

1. **GPU 感知能力**: 當使用者同時運行 ComfyUI 時，Prism 的背景 AI 任務不應該搶佔 VRAM
   - **現狀**: 目前的 Embedding 和 Batch Tagging 沒有任何 GPU 資源感知
   - **建議**: 加入 CPU fallback 選項或 GPU 佔用偵測

2. **Prompt Diff 功能**: 譜系樹 (parent_id) 應該顯示父子節點之間的差異
   - **現狀**: 已實現 `parent_id` 欄位，但前端只顯示「來自: [父標題]」連結
   - **建議**: 這是 Phase 3.7.3 (Icebox) 的功能，優先級合理

### 🧪 測試策略的務實考量 (來自 009)

文檔「009-自動化測試.txt」的測試策略建議很務實：

| 測試類型 | 適用場景 | 已實現? |
|---------|---------|--------|
| 自動化 E2E | 核心路徑 (Happy Path) | ✅ 已實現 |
| 手動測試 | 體驗與美感 | ⏳ TODO-V2.md 中有清單 |
| 探索性測試 | 亂按破壞 | ❌ 無 |

**Linus 式評論**: 你們的測試策略是正確的 - 「廚房能做菜」(API 測試) + 「服務生端菜沒打翻」(E2E 測試)。不要為了 100% 覆蓋率而過度測試 UI 細節。

### 📋 未來功能的技術債預警

根據 `007-Prism V1.4.2後的昇級方向.md`，計劃中的功能可能引入新的複雜度：

1. **RSS 雙資料庫策略**: 如果實現，需要處理跨資料庫一致性問題
2. **Telegram Bot Long Polling**: 需要額外的背景線程管理（和 Embedding 線程一樣的問題）
3. **pHash 去重**: 需要謹慎處理「微變體」誤判問題
