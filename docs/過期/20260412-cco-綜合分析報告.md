### 🟠 2.1 測試 fixture 不跑 migration —— 所有 schema 回歸測試都是假的

> **✅ 已完成 (2026-04-12 / v2.4.2)**
>
> **處理方式**：
> - `tests/conftest.py` `temp_db()` 完全重寫：建立最小 pre-migration base schema（含 `type` 欄位，以真實模擬遷移前狀態）+ 呼叫 `migrations.run_migrations(conn)` 走完整遷移鏈
> - 遷移後 schema 不含 `type`，seed 資料改用 `category_id` 直接插入
> - `sample_note_data` fixture 移除死欄位 `'type': '筆記'`
> - 新增 `tests/test_schema_regression.py`（4 個測試）：
>   - `test_notes_type_column_absent` — `Notes.type` 必須消失
>   - `test_notes_required_columns_present` — 所有 migration 加入的欄位必須存在
>   - `test_ai_columns_stripped` — AI 欄位必須不存在
>   - `test_fixture_schema_matches_migration` — fixture 輸出必須與 migration chain 一致

**檔案**：`tests/conftest.py:57-157`

```python
def temp_db():
    conn = sqlite3.connect(db_path)
    conn.executescript('''
        CREATE TABLE IF NOT EXISTS Notes (
            ...
            type TEXT DEFAULT '筆記',         # ← 手寫舊 schema，包含已刪欄位
            category_id INTEGER,
            ...
            is_system INTEGER DEFAULT 0,     # ← 真實 schema 沒有這欄
            ...
```

**Linus 評語**：
> 「測試的價值是『讓你睡得安穩』。你現在的測試是『讓你以為自己睡得安穩』。這比沒測試還糟糕——沒測試的人會緊張地手動驗證，有假測試的人連手動驗證都省了。
>
> 修法只有一個：**測試的 schema 必須與 production 走同一條 migration 路徑**。如果速度是問題，那就為 migration 加快取，而不是繞過它。」

**建議重構**：
```python
@pytest.fixture(scope='session')
def base_schema_db():
    """One-time migration run, snapshot the file, copy for each test."""
    db_path = tempfile.mktemp(suffix='.db')
    flask_app = create_app('testing')
    flask_app.config['DATABASE'] = db_path
    with flask_app.app_context():
        from app import init_db
        init_db()
    yield db_path
    os.unlink(db_path)

@pytest.fixture(scope='function')
def temp_db(base_schema_db):
    fd, db_path = tempfile.mkstemp(suffix='.db')
    os.close(fd)
    shutil.copyfile(base_schema_db, db_path)
    yield db_path
    os.unlink(db_path)
```
這樣 migration 跑一次（~50ms），每個測試 copy 一份檔案（<10ms），既快又真實。

### 🟠 2.4 `routes/notes/crud.py:487` 註解與事實矛盾

> **✅ 已完成 (2026-04-12 / v2.4.2)**
>
> **處理方式**：
> - 移除 `delete_note()` 的三條手動 cascade DELETE（`Note_History` / `Note_Tags` / `Source_Urls`）
> - 改為依賴 `init_db()` 建立時定義的 `ON DELETE CASCADE` FK 約束
> - 更新註解為 `# ON DELETE CASCADE handles Note_History / Note_Tags / Source_Urls automatically`

> **✅ 已完成 (2026-04-12 / v2.4.2)**
>

**建議**：在 `Config.PRODUCTION` 模式關閉這個 fallback，並在 `DEPLOYMENT.md` 文件化。
