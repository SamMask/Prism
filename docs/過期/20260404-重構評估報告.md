# Prism 重構評估報告：Go vs. Rust

> **報告日期**：2026-04-04  
> **評估範圍**：從目前的 Python (Flask) 後端架構遷移至 Go 或 Rust，針對效能、開發體驗、AI 生態整合及後續維護之影響分析。

---

## 1. 執行摘要
目前 Prism 採用 **Python (Flask)** 作為後端，搭配 **SQLite (FTS5)** 與 **React SPA**。Python 的最大優勢在於能極其輕易地整合 AI 模型與資料科學套件（如 `sentence-transformers`, `numpy`）。
若將專案後端以 **Go** 或 **Rust** 重新撰寫，將大幅提升**執行效能**、**並行處理能力**以及**資源使用率（降低記憶體消耗）**，並簡化使用者的佈署流程（單一執行檔）。然而，最主要的挑戰將是**本地端 AI 模型推論（Sentence Transformers）的整合**。

---

## 2. 當前架構分析 (Python + Flask)
- **優勢**：開發快速、無縫整合 AI 生態圈、動態語言彈性高。
- **痛點**：
  1. **佈署與分發**：Portable 版需打包整個 Python 環境（產出檔案通常較大，目前的 Portable 版約需 80MB+）。
  2. **效能與記憶體**：啟動速度較慢，AI 模型與 Flask 佔用的記憶體較多（對於背景常駐應用而言較為吃重）。
  3. **並行處理**：Python 的 GIL 限制了多執行緒效能，處理高併發的 API 請求時不易達到極致效能。

---

## 3. Go 語言重構方案評估
Go 是為了建立高並行、雲端原生的應用而生，非常適合開發高效能的 API 伺服器。

### 優勢
1. **極速啟動與低記憶體佔用**：相比 Python，Go 應用啟動時間為毫秒級，且常駐記憶體遠低於 Python。
2. **單一執行檔（Single Binary）**：透過 `go build` 可以打包成跨平台的單一執行檔，無需 Python 環境，大幅縮減 Portable 版容量並降低使用者的安裝門檻。
3. **優異的並行性（Concurrency）**：原生的 Goroutine 非常適合處理大量背景任務（如：批次備份、資料同步、長時間等待的外部 AI API 呼叫）。

### 挑戰與生態系轉換
- **AI 模型推論困難**：要替換掉 `sentence-transformers` 是重構 Go 最痛苦的地方。需依賴 ONNX Runtime 的 Go 綁定（如 `onnxruntime-go`），或透過 CGO 呼叫 C/C++ 函式庫，設定繁瑣且容易遇到跨平台編譯陷阱。
- **Web 框架對應**：Flask 可對應為 `Gin`, `Echo` 或 Go 1.22+ 原生 `net/http`。
- **資料庫**：已有極佳的 SQLite 支援（如 `mattn/go-sqlite3`），亦可使用 `GORM` 或 `sqlc` 進行資料庫互動。

---

## 4. Rust 語言重構方案評估
Rust 提供極致的效能與記憶體安全，近年逐漸滲透進底層工具與高效能應用開發領域。

### 優勢
1. **頂尖效能與極致最佳化**：CPU 密集操作與 I/O 處理效能遠勝 Python，且無 GC (垃圾回收)，不會在此方面產生額外延遲。
2. **高安全性與嚴謹的型別系統**：編譯過了通常就不會有執行期的記憶體問題 (`any` 型別濫用在 Rust 是行不通的)，非常適合建構需要高穩定性的工具。
3. **Rust x AI 逐漸成形**：Rust 在 AI 與機器學習生態已有起色，例如 `candle` (Hugging Face 推出的 ML 框架) 或 `tch-rs` (C++ PyTorch 綁定)，在推論上可能比 Go 更好實作本地 AI 支援。

### 挑戰與生態系轉換
- **學習曲線極其陡峭**：開發速度將顯著低於 Python 或 Go。
- **編譯時間長**：大型專案需要更久的編譯時間，開發與除錯迴圈（Developer Loop）較長。
- **Web 框架對應**：Flask 對應為 `Axum` 或 `Actix-web`。
- **架構設計改變**：需要以所有權 (Ownership) 的思維來處理共享狀態（如背景 AI Worker 或 Database Connection Pool）。

---

## 5. 關鍵技術難點比對：AI 與向量運算
目前 Prism 重度依賴 `sentence-transformers` 做語意搜尋，這是重構的最大路障。
- **若維持現狀 (Python)**：直接引入套件即可使用，最舒適。
- **Go 路線**：Go 在機器學習庫較貧乏。解決方案可能是將 AI 推論剝離成微服務（導致架構複雜，失去單一檔案優勢），或者將模型轉成 ONNX 格式，使用 CGO 呼叫 ONNX Runtime。
- **Rust 路線**：可以使用 `rust-bert` 或 HuggingFace 的 `candle` 框架載入模型進行推論，這條路在純編譯語言中比 Go 更有可行性，不過依然會增加打包與跨平台建置的難度。

## 7. 結論與重構建議

1. **短期不建議純後端重構**：若 Prism V2 的首要任務是推進 UI/UX (如 `20260315-claude4.6-綜合報告.md` 所述) 與優化現有流程，Python 的疊代速度仍是無可取代的最佳選擇。
2. **如果核心目標是「極致輕量化與桌面版分發」：推薦 Go**
   - 如果我們能將**語意搜尋的 AI 推論能力**拔除，或者改用純雲端 API (如 OpenAI 產生 Embeddings)，Go 會是最適合打造「輕量、高速、單一執行檔」的 Prism 後端工具。這能立刻殺出桌面應用的血路。
3. **如果目標是「兼顧本地 AI 執行與極致效能」：考慮 Rust (或混合架構)**
   - 如果堅持要在本地無連網狀態下做 Semantic Search，Rust 的 `candle` 或 `rust-bert` 是一個可以打造出單一執行檔帶本地推論的方案。但這會大幅增加開發成本，不適合一般專案或小團隊的維護節奏。
   - **替代混合方案**：後端大部分業務邏輯用 Go/Rust 撰寫，而把 AI 依賴單獨打包成一個微型的 Python 執行檔或使用 ONNX 執行庫，兩者透過 IPC/gRPC/API 溝通。

**下一步行動建議**：
在尚未解決前端 UI/UX 改善與技術債之前，貿然進行後端語言重整風險極高。建議**先專注執行之前的 UX 綜合報告**所列之待辦事項。若未來對於 Portable 版本的容量與常駐記憶體有嚴格要求，屆時再來考慮 Go 或 Rust 替換 Flask 的具體可行性。

## 8. 架構轉型與 API 中心化評估 (2026-04-04 補充更新)

基於最新討論決策：**「拔除所有本地 AI 語意與標籤功能，將 Prism 轉型為純粹的筆記軟體與 Headless KMS (知識中樞 API)」**，此方向對於專案架構有極為深遠且正面的影響：

### 1. 完美排除 Go/Rust 重構的最大路障
先前評估中，依賴 Python 的 `sentence-transformers` 是重構最大的阻力。拔除此依賴後：
- **Go 語言** 瞬間成為絕佳的選擇。可用最少的心力打造出處理 SQLite (FTS5) 搜尋與提供 RESTful API 的高效能後端。
- **打包體積與啟動速度** 將獲得徹底解放，能夠輕易將整個後端編譯成僅 10~20 MB 左右的單一執行檔，達成極致的 Portable 體驗。

### 2. 化身為 Agentic RAG 之基礎設施
不再強求應用程式自身具備智慧能力，而是轉向提供穩定、極速的 API 供外部 Agent (如 MuseForge 專案中的 LLM) 呼叫。這與當今最前沿的「代理式檢索」架構更為契合。Prism 將專心處理：
- 標準 Markdown 筆記的 CRUD 存取。
- 基於 SQLite FTS5 的極速全文檢索。

### 3. 短期效益：Python 架構大幅瘦身
即使首波不更換開發語言，單純拔除 AI 功能也能立刻帶來巨大好處：
- 可從 `requirements.txt` 中完全移除 `numpy`、`sentence-transformers` 等肥大套件。
- 解決不同作業平台間的相容性問題，並將打包容量與常駐記憶體縮減至原本的極小比例。

**更新後結論**：
這是極具戰略眼光的現代化拆分解決方案。建議**首要步驟為清理目前 Python 專案中的 AI 依賴與程式碼**，完善 Prism 對外的標準 API 規範。未來一旦有需要，就能直接無痛以 Go 或 Rust 對後端進行輕量化重構，打造最完美的 Agent 外掛記憶大腦。
