## 一、總體判斷

1. Prism 目前已非常接近「Go primary 可作為主要產品 runtime」的狀態：`scripts/start_go_primary.ps1` 會建置/啟動 Go artifact，並以 `--enable-*` 全開方式接 `knowledge.db` 與 repo data dir（`scripts/start_go_primary.ps1:49-70`）；Pi setup 也把 Caddy 全量 proxy 到 `127.0.0.1:5004` 的 `prism-go-primary.service`（`deploy/raspberry_pi/setup.sh:75-105`）。
2. 最大風險不是 Go 主幹完全沒做完，而是「已宣稱 live/default Go primary」之後，仍有幾個前端可觸發的 Python-only API 沒有 Go 對應路由；在 Go primary 下會變成 404 或 SPA fallback，造成實際 UX 斷點。
3. 已確認的漏接點包含：AI prompt metadata extraction、長文自動分離/還原、更新檢查，以及 PromptBuilder 的 `/static/config/wizard_options.json` 載入路徑。
4. Go backend 的核心 notes CRUD、search、uploads、attachments、cleanup、import/export、server/system 多數已有 Go implementation 與 phase tests，但現有測試沒有針對「frontend 實際呼叫清單 vs Go mux」做總表驗證，因此這些漏接沒有被擋下。
5. 文件狀態有明顯分裂：`docs/TODO.md`、`README.md`、`DEPLOY-PI.md` 宣稱 Go primary 已是 live/default；但 `docs/contracts/go-primary-route-ownership-manifest.json` 仍標 `production_runtime_owner: "python"`，`Prism_Go_模組逐步重構計劃報告.md` 後段仍保留 retained-Python 結論，`docs/API_REFERENCE.md` 也有多處 Go shadow / Python owner 舊語句。
6. 根目錄本身有 local runtime 產物與個人歸檔痕跡；其中多數被 `.gitignore` 忽略，但已追蹤的 `resources/python-embed.zip`、`resources/wheels/pillow-12.0.0-...whl` 與 T045「embedded Python / Pillow runtime removed」語義衝突。
7. 我建議現在進入修復階段，但主線要非常窄：先補 Go primary 下前端會打到的 missing API / static config，再補 route parity regression；不要進大型 redesign 或 T046 Python source 刪除。

## 三、Go backend / API contract 檢查

### 已確認有 Go 實作或路由的部分

- Go mux 註冊了核心 read/write/API surface：notes、categories、tags、attachments、system stats/maintenance、server backup/version/logs、prompt/wizard options、cleanup、export/import、upload/upload-url/upload-delete，以及 embedded SPA fallback（`go-shadow/main.go:176-219`）。
- Product startup 會用 production guard + 全部 `--enable-*` flag 啟動 Go artifact，包含 notes/category/tag/attachments/upload/cleanup/import-export/server-system（`scripts/start_go_primary.ps1:49-70`）。
- Pi setup/systemd unit 也以 `prism-go-primary.service` 啟動同一組 flags，並將 Caddy reverse proxy 到 Go port 5004（`deploy/raspberry_pi/setup.sh:75-105`）。
- Go static handler 對 unknown `/api/*` 回 JSON 404，避免 API fallback 成 SPA；對 `/static/uploads/<file>` 有 data-dir 範圍防護（`go-shadow/main.go:1266-1338`）。
- 測試檔案存在並覆蓋大量 Go phase：`tests/test_go_primary_t011_t012_notes.py`、`tests/test_go_primary_t019_attachments_metadata.py`、`tests/test_go_primary_t020_t023_files_uploads.py`、`tests/test_go_primary_t028_t031_import_export.py`、`tests/test_go_primary_t032_t035_server_system.py`、`tests/test_go_primary_t036_t038_static_security_workflow.py` 等。

### 疑似漏接的部分

- `POST /api/upload/extract-prompt`：Python route 有，Go mux 無，前端仍呼叫。
- `GET /api/notes/<id>/check_separation`、`POST /api/notes/<id>/separate`、`POST /api/notes/<id>/restore`：Python route 有，Go note router 無對應 branch；`/restore/<history_id>` 不是同一個功能。
- `GET /api/system/check-update`：Python route 與 API docs 有，Go mux 無。
- `/static/config/wizard_options.json`：前端仍依賴，但 Go primary dist 不包含該檔，Go static handler會回 SPA HTML 而非 JSON。
- `GET /api/system/go-read-routing`：API docs 仍列為 Python-owned Phase 19 狀態 endpoint（`docs/API_REFERENCE.md:716-750`）；Go primary product path是否仍需要此 endpoint 不確定，但目前 Go mux 沒有。

## 四、Frontend / UX 檢查

### 會影響實際使用的 UX 問題

- 長文 note 儲存後自動分離失敗會被靜默吞掉：`useNoteForm` 在內容超過 5000 字時呼叫 `api.separateContent(savedNoteId)`，catch 區塊是空的（`frontend/src/hooks/editor/useNoteForm.ts:122-124`）。Go primary 缺該 route 時，使用者只看到「筆記已建立/已更新」，但內容沒有分離。
- Prompt extraction 功能會失效但不明講：`usePromptExtraction` 呼叫 `api.extractImagePrompt()`，失敗時只把 `hasAIPrompt` 設為 false（`frontend/src/hooks/editor/usePromptExtraction.ts:20-25`）。Go primary 缺 route 時，UI 會像「圖片沒有 prompt」而不是「功能不可用」。
- Settings 更新檢查會把 404 顯示成「無法連接更新伺服器」（`frontend/src/components/settings/UpdateSection.tsx:20-33`），錯誤提示會誤導使用者。
- PromptBuilder wizard options 載入 `/static/config/wizard_options.json`（`frontend/src/hooks/usePromptBuilder.ts:216-223`），Go primary 下可能拿到 `index.html` 而非 JSON，造成 wizard options 空白或錯誤。

### 錯誤提示 / loading / empty state 問題

- 大部分核心 settings / data manager 有 loading 或 toast，但上述漏接 API 的錯誤訊息不夠精確。
- `separateContent` 的 silent catch 是最高優先 UX 問題，因為它掩蓋了資料處理失敗。
- `extractImagePrompt` 的 silent fallback 可能合理避免打擾，但在 Go primary migration 完成前會掩蓋 contract 漏接；建議至少在 dev/log 或 feature flag 層可見。
- `UpdateSection` 有 loading state，但沒有顯示 HTTP status / backend route missing 類錯誤。

### 不建議現在處理的純美化項目

- 不建議做大型 redesign、卡片視覺重排、設定頁資訊架構重做。
- 不建議新增 server-side UI preference persistence、collections schema、AI/embedding、timeline、協作、plugin。
- 不建議為了修 `/static/config/wizard_options.json` 而改整個 PromptBuilder 架構；最小修法是統一路徑到既有 `/api/wizard-options` 或讓 Go 明確服務 config JSON。

3. 新增 PromptBuilder static/config regression：Go primary 下 `/static/config/wizard_options.json` 不應回 HTML；或 frontend 改用 `/api/wizard-options` 後測該路徑。
4. 執行順序建議：targeted pytest for new tests → `go test ./...` → `pytest tests/ -v` → frontend typecheck/build → 必要時 Playwright smoke。

## 七、建議修復順序

1. 先保資料安全：不要刪 `knowledge.db`、`static/uploads`、`docs/attachments`、`backups`、Pi/live evidence；先補 regression，避免修漏接 API 時誤動 production data。
2. 再修整合漏接：先補 Go primary 下前端會打到的 missing surface：`upload/extract-prompt`、long-content separation/restore、`system/check-update`、PromptBuilder wizard config path。
3. 再補測試：新增 frontend-to-Go route contract test 與 targeted Go HTTP tests，再跑 targeted pytest / Go tests / full pytest。
4. 再整理根目錄：先處理 tracked 且明顯與 T045 衝突的 `resources/python-embed.zip`、Pillow wheel、root `package-lock.json`、Reddit HTML；忽略中的本機產物只做 local cleanup，不寫進 commit。
5. 最後 UX 微調：修 silent fail/error wording，讓長文分離、prompt extraction、update check 在功能不可用時有清楚提示。

## 八、禁止現在做的事

- 不要新增功能或引入 AI/embedding/協作/plugin/timeline/社群功能。
