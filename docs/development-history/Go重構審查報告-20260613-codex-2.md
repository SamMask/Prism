# Prism Go 重構收尾審查報告

## 一、總體判斷

目前 Prism 已接近可用的 Go primary 狀態：根據 `go-shadow/main.go:188-232` 的 route registry、`frontend/src/services/api.ts:205-836` 的 frontend API client，以及 `docs/TODO.md:5-13` 的近期摘要，T046-T052 的主要 Go API 漏接已被收斂。這次審查沒有找到明確的 frontend product API 呼叫「完全沒有 Go handler」的 P1 斷點；先前常見的 `/api/upload/extract-prompt`、長文分離、`/api/system/check-update`、PromptBuilder wizard options 都已在 Go route 或 frontend 呼叫中出現。最大風險不在新功能，而在 T053 尚未完成：Python source、legacy static/templates、pytest fixture 與 schema regression 還在作為 source/dev/test context 留存，不能直接刪除。文件 current truth 仍有幾處內部矛盾，特別是 `docs/ARCHITECTURE.md:158-163`、`docs/CONTRACTS.md:8`、`docs/FRONTEND-REDESIGN-PLAN.md:95-122` 還保留「Python normal owner / Go target」語氣，和目前 Go primary live/default runtime 不一致。測試面已有純 Go e2e 與 Go unit 補強，但本輪審查沒有執行測試，所以不能把 `docs/TODO.md:8` 的歷史 pass 數字當成本輪驗證結果。根目錄與 repo 內仍有 runtime data、ignored cache、build artifact、legacy source context 混在同一工作樹中；清理時最大的 P0 風險是誤刪 `knowledge.db`、`static/uploads/`、`docs/attachments/`、`docs/notes/` 或備份。建議現在進入修復階段，但主線應限縮為 T053 guardrail、文件 truth 收斂、測試驗證與安全清理，不應開新功能或 redesign。

## 三、Go backend / API contract 檢查

已確認正常或已接上的部分：

- Go primary route registry 已涵蓋核心 API：notes、categories、tags、attachments、upload、cleanup、export/import、settings/system、server dashboard、prompt options、wizard options，位置在 `go-shadow/main.go:188-232`。
- Frontend 主要 API client 對應 Go route：notes CRUD、duplicate/reorder、categories/tags、upload/url/extract-prompt、attachments、long content separation、cleanup、export/import、pin/archive、history、system、port config、CSRF、server dashboard，位置在 `frontend/src/services/api.ts:205-836`。
- 先前漏接的 T047-T050 items 在 route 與 contract 裡已能對應：`POST /api/upload/extract-prompt` 見 `go-shadow/main.go:228`，long content separation 見 `frontend/src/services/api.ts:457-487` 與 Go notes detail route `go-shadow/main.go:232`，`GET /api/system/check-update` 見 `go-shadow/main.go:196`，wizard options API 見 `go-shadow/main.go:213-214` 與 `frontend/src/hooks/usePromptBuilder.ts:219`。
- Go primary 啟動腳本已把產品寫入/檔案/匯入/伺服器操作 gate 打開：`scripts/start_go_primary.ps1:58-70`；Pi service 也走 Go artifact 與相同 gate：`scripts/go_primary_pi_live_ops.ps1:267-271`, `deploy/raspberry_pi/setup.sh:104-112`。
- Go 端 CSRF gate 已存在：`go-shadow/main.go:1437-1499`；API 文件也寫明 Origin/Referer 行為與 public internet 限制：`docs/API_REFERENCE.md:41-45`, `README.md:157-159`。

疑似漏接或需要再驗證的部分：

- 本輪沒有找到明確 frontend product API 呼叫缺少 Go handler 的 P1 問題；但這是靜態比對結果，未跑瀏覽器 e2e。
- `/api/system/go-read-routing` 在 `docs/API_REFERENCE.md:51`, `docs/API_REFERENCE.md:738`, `docs/API_REFERENCE.md:912` 被標成 legacy Flask source-only，不是 Go primary product API。若 T053 後仍保留文件入口，需明確處理其歸宿。
- `docs/contracts/go-primary-route-ownership-manifest.json` 的 `go_candidate` 歷史欄位仍會製造「live media cleanup remains Python-owned」的錯覺；這是 contract 文件問題，不是目前已確認的 handler 缺失。

舊 backend / mock / stub / placeholder 殘留：

- Python backend source 仍存在於 `app.py`, `routes/`, `utils/`, `db.py`, `config.py`, `templates/`, `static/`，但 current docs 將其定位為 T053 前的 legacy/dev/test context；不應在未完成 T053 前直接刪除。
- `routes/system.py` 與 `app.py` 仍保留 legacy system/read-routing 類入口；依 `docs/API_REFERENCE.md:912`，這是 T053 要處理的 Flask source-only 殘留。
- `go-shadow/main.go` 的 TODO / stub / not implemented 搜尋未看到明顯空 handler；但仍需以 go test 與 e2e 驗證，不可只靠字串掃描。

需要補測的 API：

- notes CRUD + FTS/card fields/text attachment body search：Go unit 已見 `go-shadow/main_test.go` 相關測試，但需本輪重跑。
- upload、attachment delete/media cleanup、import/export、cleanup、settings/system、CSRF：Go route 已存在，需用 `go test ./...` 與 pure Go e2e 補最新證據。
- `/api/server/*` local-only 行為：已有 API 文件與 Go route，需保留 localhost/Origin guard 的回歸測試。

## 四、Frontend / UX 檢查

會影響實際使用的 UX 問題：

- 長文自動分離失敗已不再是完全 silent fail：`frontend/src/hooks/editor/useNoteForm.ts:125-126` 會顯示「筆記已儲存，但長文自動分離失敗」warning。這是已改善項。
- 更新檢查 404 也已被特別處理：`frontend/src/components/settings/UpdateSection.tsx:25-33` 會顯示 Go primary 尚未提供更新檢查 API 或無法連接更新伺服器。這是已改善項。
- PromptBuilder wizard options 載入失敗疑似只 `console.error`：`frontend/src/hooks/usePromptBuilder.ts:219-252`。如果 API 失敗，畫面是否有明確錯誤狀態不確定，需瀏覽器驗證。
- Settings 頁 `GET /api/test` 失敗目前主要在 console 記錄：`frontend/src/pages/SettingsPage.tsx:89-104`。如果 health check 失敗，使用者可能只看到狀態未更新，不一定知道是連線問題。
- 圖片 prompt extraction 的 metadata check catch 是空處理：`frontend/src/hooks/editor/usePromptExtraction.ts:24`；真正 extract 失敗最後可能只顯示「圖片中未找到 AI 提示詞」：`frontend/src/hooks/editor/usePromptExtraction.ts:55`。這可能把 backend failure 和真的沒有 metadata 混在一起。

錯誤提示 / loading / empty state 問題：

- 已有局部 loading/error state，例如 PromptBuilder hook 有 `isLoading` 與 `error` 狀態；但直接 fetch 的錯誤是否都能進到畫面訊息不一致。
- 本輪未啟動 frontend dev server，也未用瀏覽器實測 mobile/responsive、empty state 或 loading state，所以這些只能列為「疑似需驗證」，不能宣稱已通過或失敗。

不建議現在處理的純美化項目：

- 不建議重開 `docs/FRONTEND-REDESIGN-PLAN.md` 的大型 redesign。
- 不建議調整色彩、字體、layout density、資訊架構等非 blocking 美化。
- 不建議新增 AI、timeline、collaboration、plugin、community 或 collections schema。

## 五、測試與驗收缺口

現有測試狀態：

- `tests/` 目前有大量 pytest 檔案；本輪盤點到 `test_*.py` 約 105 個。
- 純 Go e2e acceptance 已存在：`tests/test_go_primary_e2e_pure_go_acceptance.py` 會建 fresh Go-created DB、啟動 Go runtime、跑 `scripts/go_primary_full_workflow_smoke.py`，並檢查測試自身不 import legacy app/routes/migrations/services/config/db。
- Go unit 測試已在 `go-shadow/main_test.go` 覆蓋 schema/migration、notes/search、delete cleanup、upload URL security、pin/archive/duplicate/reorder、tags merge、history、CSRF 等面向。
- Legacy Python/pytest 測試仍存在，這是 T053 前保留的 source/dev/test context。`docs/T053-test-disposition-plan.md:154-189` 明確指出剩餘前置門檻是 `test_schema_regression.py` 轉到 Go schema truth。
- 本輪未執行 `pytest` 或 `go test`，因此不宣稱測試通過；`docs/TODO.md:8` 的 pass 數字只能視為文件中的歷史驗證紀錄。

缺少的核心測試或驗收證據：

- T053 前仍需要一次明確的 current-run `cd go-shadow && go test ./...`。
- T053 前仍需要 targeted pytest，至少包含 pure Go e2e、Go primary route coverage、schema regression disposition、import/export、attachment/media cleanup。
- 需要確認 pure Go e2e 在本機 Go CLI 存在時不是 skip；`tests/test_go_primary_e2e_pure_go_acceptance.py:100-108` 目前有 skip gate。
- 需要確認 legacy pytest 中哪些 test 應轉 Go、哪些應刪、哪些只保留 migration/history evidence；不可用測試數量代替 contract coverage。

最小建議補測清單：

1. `cd go-shadow && go test ./...`
2. `pytest tests/test_go_primary_e2e_pure_go_acceptance.py -v`
3. `pytest tests/test_go_primary_t046_t050_frontend_route_coverage.py -v`
4. `pytest tests/test_schema_regression.py -v`，但先依 T053 plan 改成 Go schema truth 後再作為 deletion gate
5. 針對 frontend direct fetch failure path 補最小 UI/handler 測試或手動瀏覽器 smoke

## 七、建議修復順序

1. 先保資料安全：確認 `knowledge.db`、uploads、attachments、notes、backups 不在任何清理清單內，必要時先做備份與只讀 inventory。
