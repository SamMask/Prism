# 1230-審核報告

## 核心判斷
❌ **不值得做（以目前狀態）**：如果您打算在目前基礎上繼續堆砌功能。
✅ **值得做**：如果您願意先花一週時間清理核心資料結構與非同步邏輯。

這個專案目前處於「能跑但脆弱」的典型 V1.5 狀態。Schema V2 有正確的想法（如 `AI_Tasks`），但程式碼 (`crud.py`) 卻還停留在 V1 的補丁思維中掙扎。

## 關鍵洞察

### 1. 資料結構：雙重事實 (Double Truth) 的災難
**現狀**：`Notes` 表同時保留 `type` (字串) 和 `category_id` (外鍵)。`crud.py` 中的 `get_category_id_by_name` (L23-41) 是一個醜陋的補丁，試圖在應用層同步這兩個事實。
**Linus 觀點**："Bad programmers worry about the code. Good programmers worry about data structures."
**問題**：只要這兩個欄位並存，資料不一致就是時間問題。為什麼要讓程式碼去猜測 "筆記" 對應哪個 ID？這是資料庫的工作。

### 2. 複雜度：`get_notes` 的意大利麵條
**現狀**：`crud.py` 的 `get_notes` (L133-298) 是一個長達 160 行的巨型函式，混合了：
- 參數解析
- SQL 動態組裝 (String concatenation hell)
- 特殊業務邏輯 (封存、FTS5 清洗)
- JSON 解析
**Linus 觀點**："如果你需要超過 3 層的縮排，你就已經完蛋了。"
**問題**：這個函式已經無法維護。想加一個過濾條件需要改動三個地方。

### 3. 風險點：脆弱的非同步機制
**現狀**：`_queue_embedding_update` (L44) 在 WSGI 請求生命週期內啟動 `ThreadPoolExecutor`。
**Linus 觀點**："我們不破壞使用者空間！" -> "我們不寫會在伺服器重啟時丟失資料的程式碼！"
**問題**：
1. Web Server (如 Gunicorn/uWSGI) 重啟或回收 Worker 時，這些執行中的 Thread 會被無情這殺。
2. Schema V2 明明定義了 `1.3 AI_Tasks`，為什麼不用？這是典型的「過度設計了架構但實作懶惰」。

---

## The Linus Way: 修正方案

### 第一步：淨化資料結構 (The Purge)
**行動**：
1. **Kill `Notes.type`**。現在。立刻。
2. 執行一次性 Migration：確保所有 `Notes` 都有 `category_id`。如果找不到，歸類到 `Default/Uncategorized`。
3. 刪除 `Notes.type` 欄位。
4. 刪除 `get_category_id_by_name` 及其所有引用。
**結果**：消除了所有同步邏輯。程式碼減少 50 行，Bug 減少 50%。

### 第二步：實作真正的任務隊列
**行動**：
1. 在 `create_note` 和 `update_note` 中，不要 `submit` 到 ThreadPool。
2. 改為 `INSERT INTO AI_Tasks (task_type, payload, status) VALUES ('embedding', json_object('note_id', ?), 'pending')`。
3. 寫一個獨立的 Worker Process (或是簡單的 Cron job / Check script) 來消化 `AI_Tasks`。
**結果**：任務持久化。伺服器崩潰也不會丟失 Embedding 請求。

### 第三步：重構查詢邏輯
**行動**：
1. 將 `get_notes` 的 SQL 組裝邏輯提取為 `NoteQueryBuilder` 類別或一組純函式。
2. 不要在 Python 中做 `parse_tags_json`。SQLite 的 JSON 支援已經夠強大，直接返回前端需要的結構，或者確保 DB 輸出的 JSON 已經是乾淨的。

### 程式碼審查摘要 (crud.py)
| 項目 | 評分 | 評論 |
| :--- | :--- | :--- |
| **品味** | 🟡 | 試圖用 `get_category_id_by_name` 掩蓋架構錯誤，這是壞品味。但使用了 SQLite JSON 功能是好品味。 |
| **結構** | 🔴 | `get_notes` 太過龐大且包含太多職責。`_do_embedding` 定義在函式內部很醜陋。 |
| **安全** | 🟡 | FTS5 的字串處理 (L180) 比較原始，但有意識到 DoS 風險。 |

**總結**：
別再寫 Python 補丁了。修好你的 Database Schema，程式碼自然會變簡單。

> "Design the architecture, create the data structures, and then write the code."
